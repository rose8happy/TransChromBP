# 基座模型接入路线图

> 这是 foundation 线路线解释，不是最新实验状态台账。
> family verdict、run 级记录和 terminal evidence 统一以 [docs/experiments/registry.md](../experiments/registry.md)、[docs/experiments/runs.csv](../experiments/runs.csv) 和对应 closeout 为准。

这份文档只讲一件事：

> Genos、NT v2、Caduceus、AlphaGenome 分别是怎么接进项目的，结果怎样，为什么没有变成当前默认主线。

先给一句总判断：

> 当前仓库里，“不接 foundation model 的主线”比“接了 foundation model 的支线”更稳定；foundation 线更多是高质量负结果、边界实验，或者外部 sidecar/reference。

## 1. 先把状态分三类

| 类别 | 当前有哪些 | 应该怎么理解 |
|---|---|---|
| 有 clean gain | 目前没有稳定 family 被归到这一类 | 不要把某次小 improvement 直接包装成主线晋级 |
| 有信号但不互补 | `NT v2`、`Caduceus` | 模型本体或特征有信息，但没在当前接法下转成净收益 |
| 技术跑通但不作为质量比较 | `AlphaGenome matched raw-track sidecar`、`Genos OCR benchmark` | 说明链路或模型本体可用，不代表 integration 赢了 |

## 2. 不接 foundation 时，主线长什么样

基线长这样：

```text
DNA one-hot
  -> ConvStem
  -> LocalDilatedTower
  -> optional Transformer
  -> profile/count heads
  -> bias_branch
  -> full / debiased outputs
```

也就是说，当前主线已经能在不借外部大模型的情况下，把 profile/count 做得很强。

如果你想先把这条主线看懂，再回来理解 foundation 线，建议先读：

- [无大模型基线到当前主线.md](无大模型基线到当前主线.md)

## 3. 接法一：cached summary / 轻量注入

这条线最典型的是 `Genos`。

### 3.1 为什么会想到这么接

因为 `Genos` 这类模型天然先产出 hidden states / summary，最直接的想法就是：

- 先把外部模型 frozen
- 把 summary 当额外特征喂给现有主干
- 看能不能低成本补强 profile/count

### 3.2 代码和配置入口

| 项目 | 入口 |
|---|---|
| 模型模块 | `../../vendor/transchrombp/transchrombp/models/genos_adapter.py` |
| 主模型挂载点 | `../../vendor/transchrombp/transchrombp/models/transchrombp.py` |
| cache 构建 | `../../vendor/transchrombp/transchrombp/scripts/build_genos_summary_cache.py` |
| probe | `../../vendor/transchrombp/transchrombp/scripts/run_genos_summary_probe.py` |
| 训练脚本 | `../../vendor/transchrombp/transchrombp/scripts/run_genos_cached_p0.sh`、`run_genos_cached_pilot.sh` |

### 3.3 代表实验

| 问题 | 改动点 | 代码入口 | 配置入口 | 代表 run | 关键指标 | 结论 | 详细报告 |
|---|---|---|---|---|---|---|---|
| `Genos` summary 有没有独立信号 | 先做 probe，不直接大训 | `genos_adapter.py`、`run_genos_summary_probe.py` | cached-probe 配置 | `genos_only_auc`、`concat_auc` | `genos_only AUC=0.6997`，`encoded_only AUC=0.9219`，`concat AUC=0.9018`；`count residual R²=0.0167` | 有信息，但弱、粗，而且和当前 backbone 不互补 | [reports/analysis/genos_no_positive_gain_analysis_20260323.md](../../reports/analysis/genos_no_positive_gain_analysis_20260323.md) |
| 直接把 Genos 接到训练里会怎样 | online gate / mean、cached count-only 注入 | `transchrombp.py`、`train_ddp.py` | `v2fix_genos_gate.yaml`、`v2fix_genos_mean.yaml`、`train_genos_cached_short10.yaml` | `G1`、`G2`、`P2` | `G1=0.3388 / 0.4689`，`G2=0.3387 / 0.6360`，`P2=0.3174 / 0.6037`；matched baseline `G0=0.3163 / 0.8410` | 高质量负结果；count 路径尤其容易被拖坏 | [reports/analysis/genos_and_6002_run_analysis_20260323.md](../../reports/analysis/genos_and_6002_run_analysis_20260323.md)、[reports/analysis/genos_cached_fusion_status_20260323.md](../../reports/analysis/genos_cached_fusion_status_20260323.md) |

### 3.4 为什么没成为主线

当前证据更支持这几个解释：

1. summary 太粗，保不住 profile/count 需要的局部结构。
2. 现有 backbone 已经很强，弱外部特征很难再补。
3. 注入位置一旦太靠近 count head，count 很容易先坏。

## 4. 接法二：在线 token-level 注入

这条线最典型的是 `Caduceus-PS`。

### 4.1 为什么会想到这么接

因为和 `Genos global summary` 相比，token-level hidden states 更细，理论上更可能保留局部结构。

### 4.2 代码和配置入口

| 项目 | 入口 |
|---|---|
| 模型模块 | `../../vendor/transchrombp/transchrombp/models/caduceus_adapter.py` |
| 主模型挂载点 | `../../vendor/transchrombp/transchrombp/models/transchrombp.py` |
| 训练时在线提取 | `../../vendor/transchrombp/transchrombp/training/train_ddp.py` |
| 模型配置 | `../../vendor/transchrombp/transchrombp/configs/model/transchrombp_teacher_v2_center_pool_caduceus_ps.yaml` |
| 启动脚本 | `../../vendor/transchrombp/transchrombp/scripts/run_caduceus_tutorial_ab.sh` |

### 4.3 代表实验

| 问题 | 改动点 | 代码入口 | 配置入口 | 代表 run | 关键指标 | 结论 | 详细报告 |
|---|---|---|---|---|---|---|---|
| frozen token fusion 会不会比 summary 更自然 | 在现有主干前接 `Caduceus-PS` token hidden states，做 gated residual 融合 | `caduceus_adapter.py`、`transchrombp.py`、`train_ddp.py` | `transchrombp_teacher_v2_center_pool_caduceus_ps.yaml` | matched A/B 单 seed | A 臂 `0.3132558744 / 0.8388160470`，B 臂 `0.3126602104 / 0.8466657043`；JSD 仅改善 `0.00060`，count 提升 `0.00785`，低于预设 `0.002` 推进门槛 | `near-null / marginal positive`，安全但不够强 | [docs/plan/archive/foundation/caduceus_tutorial_ab_online_fusion_20260331.md](../plan/caduceus_tutorial_ab_online_fusion_20260331.md) |

### 4.4 为什么没成为主线

因为这条线回答的是：

> token-level 融合在工程上可以安全接通。

它没有回答的是：

> 它值不值得进入主表、继续扩线。

当前答案仍然是否。

## 5. 接法三：cross-attention / residual / adapter 风格

这条线目前最典型的是 `NT v2`。

### 5.1 为什么会想到这么接

`NT v2` 相比 `Genos` 更像一个真正的 token-level sequence model，所以自然会想到：

- cross-attention
- residual head
- center-aligned token / bins summary

### 5.2 代码和配置入口

| 项目 | 入口 |
|---|---|
| 基础挂载模块 | `../../vendor/transchrombp/transchrombp/models/foundation_adapter.py` |
| 模型配置 | `../../vendor/transchrombp/transchrombp/configs/model/transchrombp_teacher_v2_center_pool_ntv2_cross_attention.yaml`、`transchrombp_teacher_v2_center_pool_ntv2_residual.yaml` |
| cache 构建 | `../../vendor/transchrombp/transchrombp/scripts/build_foundation_cache.py` |
| 训练脚本 | `../../vendor/transchrombp/transchrombp/scripts/run_ntv2_residual_short10.sh` |
| teacher cache 导出 | `../../vendor/transchrombp/transchrombp/evaluation/model_teacher_cache_export.py` |

### 5.3 代表实验

| 问题 | 改动点 | 代码入口 | 配置入口 | 代表 run | 关键指标 | 结论 | 详细报告 |
|---|---|---|---|---|---|---|---|
| `NT v2` 本体到底有没有可用信号 | 先做 smoke + probe，再决定是否值得 integration | `foundation_adapter.py`、`build_foundation_cache.py` | probe / residual 配置 | `nt_v2_repair_smoke_probe_20260404_211117` | `nt_only AUC=0.742424`，`concat AUC=0.857792`，baseline `encoded_only AUC=0.868116`；`nt_logcount_r=0.530499` 但 residual `R²` 全负 | 有独立信号，但不互补 | [reports/analysis/nt_v2_probe_20260404.md](../../reports/analysis/nt_v2_probe_20260404.md) |
| 更强 residual 接法能不能翻案 | coarse residual 与 bins16 center-aligned residual gate | `foundation_adapter.py`、`run_ntv2_residual_short10.sh` | `transchrombp_teacher_v2_center_pool_ntv2_residual.yaml` | short10 residual family | coarse residual `0.3560 / 0.7729`；bins16 `0.3588 / 0.7516`；matched no-foundation control `0.3193 / 0.8298` | measured family 已被 stop-rule 收口 | [reports/paper/paper_claim_evidence_matrix_20260326.md](../../reports/paper/paper_claim_evidence_matrix_20260326.md) |

### 5.4 为什么没成为主线

`NT v2` 当前最容易被误读成“只差一次更大训练就会翻案”。仓库里的证据并不支持这个判断。

更准确的说法是：

1. 它比 `Genos` 更像真的有独立信号。
2. 但这种信号没有在已测接法里转成净增益。
3. 所以当前 measured family 该停，不该自然扩线。

## 6. 接法四：外部 sidecar / API 对照

这条线最典型的是 `AlphaGenome matched raw-track slice`。

### 6.1 为什么会想到这么接

因为 `AlphaGenome` 更像一个外部 sequence-to-track 参考系：

- 它不一定和当前任务完全同口径
- 但它很适合做“窄而准”的外部坐标对照

### 6.2 代码和运行入口

这条线更多依赖 sidecar 脚本和外部环境，重点不在 `TransChromBP` 主模型内部。

当前更该看的是真实 closeout，而不是尝试把它塞进主干：

- [reports/closeout/alphagenome_matched_raw_track_slice_v2_closeout_20260410.md](../../reports/closeout/alphagenome_matched_raw_track_slice_v2_closeout_20260410.md)

### 6.3 代表实验

| 问题 | 改动点 | 代码入口 | 配置入口 | 代表 run | 关键指标 | 结论 | 详细报告 |
|---|---|---|---|---|---|---|---|
| AlphaGenome 能不能作为 external-coordinate sidecar 稳定跑通 | 不把它当训练主干，只做 matched raw-track slice | sidecar pipeline | `regions_k562_tutorial_matched_panel_v2.csv` 等面板配置 | `alphagenome_matched_raw_track_slice_v2_20260410` | `16` 个 loci 全部成功；每个位点 `num_tracks_used=1`；`summary.csv`、`merged_locus_totals.csv` 完整产出 | technical / sidecar `pass`，不是质量胜利，也不是 benchmark 晋级 | [reports/closeout/alphagenome_matched_raw_track_slice_v2_closeout_20260410.md](../../reports/closeout/alphagenome_matched_raw_track_slice_v2_closeout_20260410.md) |

### 6.4 为什么没成为主线

因为这条线回答的是“外部坐标对照能不能跑通”，不是“它能不能替代当前 teacher 训练主干”。

## 7. 一张总表：为什么 foundation 线都没升主线

| 模型 | 最像回答了什么 | 最终没有回答什么 |
|---|---|---|
| Genos | 证明模型本体没坏，而且 summary 有一点 accessibility 信号 | 没证明它能补当前 backbone，反而多条 recipe 把 count 拉坏 |
| Caduceus | 证明 token-level frozen fusion 可以安全接通 | 没证明收益大到值得 promotion |
| NT v2 | 证明“有独立信号”这件事比 Genos 更明显 | 没证明这种信号在已测接法里能形成 clean gain |
| AlphaGenome | 证明外部 raw-track sidecar 能稳定跑通 | 没证明它是当前任务下更优的主线模型 |

## 8. 一句话结论

如果你要非常高层地概括当前仓库里的 foundation 结论，可以直接说：

> 目前所有 foundation 线都更像“方法边界被问清楚了”，而不是“新主线已经出现了”；Genos 是高质量负结果，Caduceus 是 near-null，NT v2 是有信号但不互补，AlphaGenome 是 external sidecar/reference。
