# 2026-04-09 双机实验解耦调度设计

## 1. 背景与问题

当前 `6000 / A6000 x2` 与 `6002 / RTX 3080` 的实验调度，已经出现一个明确问题：

- 两台机器的“下一步”被写成串行依赖；
- `6000` 会等待 `6002` 的 cheap-screen 收口后再决定下一条；
- `6002` 也会被 `6000` 的正式 gate 结论牵引；
- 结果是双机没有真正并行，而是退化成“主从式排队”。

这不符合当前硬件现实：

- `6000` 的双 A6000 吞吐明显高于 `6002` 的单 3080；
- 两台机器承担的实验成本、等待代价、信息密度都不相同；
- 如果继续把它们写成互相等待，A6000 会出现不必要空转，3080 也会被非必要地打断。

因此，本次设计要解决的核心问题不是“foundation 和 non-foundation 怎么硬拆”，而是：

> 如何让 `6000` 与 `6002` 各自维护独立 backlog，按吞吐与信息价值分配任务，同时彻底去掉不必要的互相等待。

---

## 2. 目标

本设计的目标有四个：

1. 把双机关系从“主从等待”改成“并行解耦”。
2. 用“实验成本 / 信息价值 / 正式性”而不是“是否 foundation”作为主分配规则。
3. 明确什么实验默认进 `6000`，什么实验默认进 `6002`。
4. 在文档层面替换现有 charter 中会造成互相阻塞的表述，避免后续再次漂回旧模式。

非目标：

1. 不在本设计中决定某个具体新实验的全部实现细节。
2. 不把所有历史实验重新归档，只处理当前有效的调度口径。
3. 不把两台机器彻底按技术类型硬拆成“foundation-only”与“non-foundation-only”。

---

## 3. 方案比较

### 3.1 方案 A：技术类型硬拆

- `6000` 只跑 foundation / teacher / AlphaGenome / NT / Genos
- `6002` 只跑纯结构实验

优点：

- 规则直观，容易记忆。

缺点：

- 会把一些本来适合 A6000 快速做正式 gate 的 non-foundation 实验强行压回 3080；
- 也会让 `6000` 在 foundation backlog 空窗时浪费吞吐；
- 实际上不能最好地利用两台机器的速度差。

### 3.2 方案 B：吞吐优先 + 完全解耦

- `6000` 负责高成本、高价值、正式 gate、长任务、或明显受益于更高吞吐的实验；
- `6002` 负责 cheap-screen、确认性 rerun、轻量探索、工具/数据小验证；
- foundation 只是排序因素，不是硬门；
- 两台机器各自维护 backlog，不再互相等待。

优点：

- 最符合当前硬件现实；
- 最能减少 A6000 空转；
- 也能保留“新 family 可直接上 A6000 正式判定”的灵活性。

缺点：

- 需要把准入表写清楚，否则容易再次退回临时判断。

### 3.3 方案 C：完全自由调度

- 哪台空闲就跑哪台；
- 只按当下主观优先级决定。

优点：

- 灵活。

缺点：

- 最容易重新回到口径漂移、实验互相抢占、临时想法覆盖既定队列的问题。

### 3.4 结论

采用 **方案 B：吞吐优先 + 完全解耦**。

---

## 4. 新的双机分工

### 4.1 `6000 / A6000 x2`

`6000` 默认承担：

1. `full-budget` 或正式 gate run；
2. 在 `6002` 上 wall-clock 过长的实验；
3. 明显吃双卡 / 大显存 / 更高吞吐的训练；
4. 对主线方向有排队价值的新 family 首个正式 run；
5. 高信息密度、等待代价高、不应让 A6000 空着的任务。

`6000` 可以跑 foundation，也可以跑 non-foundation。  
它不再承担“跟着 6002 的 cheap result 再决定”的从属角色。

### 4.2 `6002 / RTX 3080`

`6002` 默认承担：

1. `short10 cheap-screen`；
2. 确认性 rerun；
3. 轻量 readout / head / loss / config ablation；
4. 小规模 data / tooling smoke；
5. 不值得占用 A6000 的 exploratory 尝试。

`6002` 可以跑 foundation，也可以跑 non-foundation。  
它的职责是快速淘汰、快速排序、快速确认，而不是承担最终正式裁决。

---

## 5. 解耦硬规则

### 5.1 默认不互相等待

两台机器各自维护 backlog，不等待对方 run 收口才决定自己的下一条。

### 5.2 允许建议，不允许阻塞

一台机器的结果可以给另一台机器提供晋级建议，但不能成为另一台机器发车的必需前提。

### 5.3 唯一允许的阻塞来源

只有共享前置物缺失时，才允许一台机器等待另一台机器，例如：

1. 尚未生成的 teacher cache；
2. 尚未完成的数据准备；
3. 尚未实现的 launcher 或 config；
4. 尚未验证的共同运行环境。

### 5.4 明确切断的旧关系

以下关系必须从 live 文档口径中删除：

1. `6000` 等 `6002` 的 cheap-screen 收口后再决定下一条；
2. `6002` 因 `6000` 的 gate 判负而自动停住；
3. `6002` 的正结果自动触发 `6000` 发车；
4. 两台机器共享一个串行的“唯一下一步”。

---

## 6. 双机准入表

### 6.1 默认进 `6000`

下列实验默认优先给 `6000`：

1. `full-budget` / paper-facing / family-facing 正式判定；
2. 预计在 `6002` 上耗时超过约 `6h` 的实验；
3. 预计在 `6000` 上耗时超过约 `3h`、且信息价值高的实验；
4. DDP 或更大 batch 才显著受益的训练；
5. 新 family 的首个正式 gate run。

### 6.2 默认进 `6002`

下列实验默认优先给 `6002`：

1. `short10 cheap-screen`；
2. confirmation rerun；
3. 小型结构消融；
4. 数据/工具链 smoke；
5. 不值得占 A6000 的 exploratory run。

### 6.3 允许跨边界的例外

1. 若 `6000` 空闲，而 `6002` backlog 中有明显重要且在 3080 上会拖很久的实验，可直接提到 `6000`。
2. 若某个 foundation 实验本质上只是轻量 API / cache / loader smoke，也可放在 `6002`。
3. 若某个 non-foundation 实验属于新的主线候选且需要正式 gate，也可直接进 `6000`。

### 6.4 明确禁止的低效模式

1. 为了“保持同步”而做镜像 run；
2. `6000` 空等 `6002`；
3. `6002` 因 `6000` 判负而自动停住；
4. 因某条线以前在 `6002` 跑过，就永远把它锁死在 `6002`。

---

## 7. 当前 backlog 的新拆法

### 7.1 `6000` 当前 backlog

按本设计，`6000` 当前应独立维护自己的 backlog，不再等待 `6002 U-Net-lite r4`。

当前优先级建议为：

1. `AlphaGenome matched raw-track slice`
   - 与 `6002` 当前 run 完全解耦；
   - 6000 上的 AlphaGenome 环境与 API smoke 已具备；
   - 信息价值高，且不再回到已判负的 adapter family。

2. 下一个真正新的高价值正式实验
   - 例如新的 `A6000-only readout family`
   - 或新的 `teacher / init` 范式
   - 但必须是新 family，而不是回头补 `msdec_v1` 或 `teacher-distill tutorial`

3. 若上述训练还未 ready
   - 允许先做 foundation 侧轻量准备任务；
   - 例如 matched loci 生成、对照脚本、cache 预备；
   - 但默认不让 `6000` 空着。

### 7.2 `6002` 当前 backlog

`6002` 按自己的 cheap backlog 独立推进：

1. 跑完当前 `U-Net-lite r4`
2. 独立完成 `U-Net-lite v1` 的 rigor closeout
3. 若判负，继续下一个 cheap readout / structure 变体
4. 若暂无 ready 的新 readout 变体，再转轻量外部对照或小型数据/工具验证

### 7.3 当前直接执行含义

按本设计，此刻的默认动作应是：

1. `6000` 立即准备并启动自己独立的下一条高价值任务，首选 `AlphaGenome matched raw-track slice`
2. `6002` 继续跑完 `U-Net-lite r4`
3. 两边并行推进，不互相卡住

## 8. 文档替换策略

### 8.1 需要替换的旧口径

当前最需要替换的是现有 charter 中造成耦合的表述，而不是实验事实本身。

必须删除或改写的口径包括：

1. `6000` 的下一条默认要等 `6002` 收口；
2. `6002` 的结果是 `6000` 的必要晋级门；
3. 某台机器的负结果会自动改写另一台机器的默认下一步。

### 8.2 新 charter 结构

建议把当前双机 charter 重写为四段：

1. 双机分工原则；
2. 解耦硬规则；
3. 准入表与例外；
4. 当前 `6000` backlog / 当前 `6002` backlog。

### 8.3 `TRACKING.md` 的写法

`TRACKING.md` 顶部 live 状态应改为并列结构：

1. `6000` 当前在做什么，下一条是什么；
2. `6002` 当前在做什么，下一条是什么；
3. 不再写成“先等 A，再决定 B”的串行模板。

### 8.4 预期实施产物

本 spec 被批准后，应实施到：

1. `docs/plan/2026-04-09_dual_machine_experiment_charter.md`
2. `TRACKING.md`
3. 必要时补一份新的 handoff / report，说明为什么旧的互相等待口径被替换

---

## 9. 测试与验证要求

这次不是代码功能设计，而是调度与文档设计。实施完成后，验证重点应是：

1. live 文档中不再残留“双机互相等待”的默认口径；
2. `6000` 与 `6002` 的 backlog 被并列写出；
3. 当前 run 状态与远端实机状态一致；
4. 若启动了新后台任务，`TRACKING.md` 同轮写入 run name、机器/GPU、日志路径、预计结束时间与下一步。

---

## 10. 一句话结论

双机实验调度应从“按技术标签硬拆或互相等待”，改成“按吞吐、成本、正式性分配，并让 `6000` 与 `6002` 各自维护独立 backlog”。在当前时点，这意味着 `6000` 不应继续等待 `6002`，而应立即启动自己独立的高价值任务；`6002` 则继续完成自己的 cheap-screen 队列。
