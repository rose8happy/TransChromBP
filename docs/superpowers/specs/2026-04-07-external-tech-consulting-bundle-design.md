# 外发包改版设计：技术路线咨询主入口（2026-04-07）

## 背景

当前外发包已经有一版较完整的 `2026-04-05` 主包，并在后续补了 `post-20260406` 的 delta。

但本轮用户明确把外发主目标改成：

1. 技术路线咨询，而不是 reviewer 式论文收口；
2. 问题重点是：
   - 我们的模型还能怎样提升；
   - AlphaGenome / U-Net 风格是否可能提升；
   - ATAC 领域常用哪些指标，我们该补哪些；
   - 预训练基因组大模型官方如何做下游 ATAC/可及性任务，为什么我们当前接入没效果，接下来该怎么做；
   - 应怎样与 AlphaGenome 做对比。

同时，用户还要求把已被排除的旧叙事
`Transformer 接入可能和 bias 机制连接存在问题`
降级，不再让它占据外部模型的主要注意力。

## 目标

在不重做整套证据包的前提下，把当前外发包改成：

- **技术路线咨询为主入口**
- **paper/reviewer 约束为副轨**
- **已排除旧叙事只保留一句背景说明**

## 非目标

- 不重写全部历史证据文件；
- 不删除所有与 paper-facing 收口有关的内容；
- 不在这一轮就为技术路线问题补新的实验；
- 不把外发包重新扩成“大而全项目总档案 2.0”。

## 总体方案

采用“双轨结构，但技术路线咨询为主入口”的方案。

### 主轨：技术路线咨询

外部模型优先回答：

1. 模型下一步最有潜力的提升路线；
2. AlphaGenome / U-Net 式架构是否值得借鉴；
3. ATAC 任务中应补哪些指标；
4. 预训练基因组大模型为何当前接入没显出效果，以及下一步该怎么做；
5. 应怎样设计 AlphaGenome 对比。

### 副轨：论文/审稿约束

现有 paper-facing 事实和 stop-rule 不删除，但只用来约束技术建议：

- 避免外部模型提出完全脱离当前证据的建议；
- 避免再次把注意力拉回已降级的旧 shortcut/bias 误会；
- 保证外部建议仍能服务后续论文、答辩与审稿防守。

## 文件策略

### 1. 重写 `00_readme_upload_order_and_prompt.md`

职责改为：

- 当前外发包的主入口；
- 说明这是“技术路线咨询版”而不是 reviewer 收口版；
- 给出第一轮推荐上传顺序；
- 提供新的技术路线咨询 prompt。

需要修改的点：

- 第一轮默认上传顺序应包含旧主包 + delta + 新的技术问题总览文件；
- prompt 主问题改成技术路线咨询；
- reviewer 问题退到次级约束，而不是开场主问题。

### 2. 新增 `09_technical_questions_for_external_models.md`

职责：

- 把本轮真正想问外部模型的问题集中写清；
- 明确优先级；
- 明确“不受当前实现约束，优先判断最有潜力方向，即使代价较大”；
- 防止外部模型在多个文件之间自己猜主问题。

计划包含的主题：

1. 我们的模型还能怎样提升；
2. AlphaGenome / U-Net 风格是否值得；
3. ATAC 领域指标应怎样补；
4. 预训练基因组大模型官方下游做法、我们当前失效可能原因、下一步路线；
5. 如何与 AlphaGenome 做公平且有解释价值的对比。

### 3. 更新 `01_project_state_claims_and_open_questions.md`

职责不变，但“当前最需要外部模型回答的问题”这一节改成技术路线优先。

需要明确写出：

- 旧 `Transformer/bias 连接问题` 已被降级，不再是当前主问题；
- 当前 foundation stop-rule 是约束，不是主问题本身；
- 当前更需要的是“下一代提升路线判断”。

### 4. 保留并微调 `08_post_20260406_delta.md`

保留它的 stop-rule 与 runtime note 作用，但让它服务于新的主目标：

- 说明为什么当前 measured family 已默认停表；
- 说明外部模型现在该讨论的是“下一代不同方向”，而不是“旧 family 再怎么微调”。

### 5. 暂不大改 `02-05`

默认把 `02-05` 继续当作证据底座使用。

只有在检查时发现某份文件仍大篇幅强化旧误会叙事，才做局部降噪。

## 新 prompt 的组织原则

外部问题按下面 5 组组织：

1. 模型还能怎样提升；
2. AlphaGenome / U-Net 风格是否值得；
3. ATAC 领域指标该怎样补；
4. 预训练基因组大模型为什么当前接入没显出效果；
5. 我们应怎样与 AlphaGenome 对比。

同时加一个硬约束：

> 不要把注意力放在“Transformer 和 bias 机制的连接可能有 bug”这个旧怀疑上；当前它只应被视为已降级的历史误会，而不是当前主问题。

## 风险与处理

### 风险 1：外部模型又被旧 shortcut / bias 叙事吸走

处理：

- 在 `00` 和 `09` 中显式降级；
- 在 `01` 的开放问题里不再把它放主位；
- 如有必要，对 `03` 做局部降噪。

### 风险 2：外部模型只给 reviewer 建议，不给技术路线建议

处理：

- `00` 明确说明此次不是 reviewer 主任务；
- `09` 把技术问题写成主问题清单；
- reviewer/paper 只保留为副轨约束。

### 风险 3：外部模型给的建议脱离我们当前证据

处理：

- 保留 `08` 的 stop-rule 和 runtime note；
- 保留 `02-05` 作为事实底座；
- prompt 中要求其在当前证据约束下判断下一代方向。

## 验收标准

完成后的外发包应满足：

1. 外部模型一眼能看出本轮主目标是“技术路线咨询”；
2. 旧 `Transformer/bias` 误会不再抢主注意力；
3. AlphaGenome / U-Net、指标、预训练基因组大模型下游接法、AlphaGenome 对比都被明确提出；
4. 现有证据和 stop-rule 仍在，能约束建议不跑偏；
5. 用户可以直接按第一轮上传顺序发出去，而不需要再手工解释本轮重点。

## 本轮实施边界

在用户审阅并确认这份 spec 之前，不继续改外发包正文文件。
